# 開發紀錄 - 2026-04-13

## 今日開發重點：Data Scope 過濾機制統一化與 RBAC 架構強化

今日對系統核心的資料存取控制（Data Scope）進行了大規模重構，將分散在各個 Service 的過濾邏輯統一抽離，並補齊了組織架構與班級學員名單的權限漏洞。

### 1. 統一 Data Scope 工具函式 (`server/utils/rbac/scopes.ts`)

為了解決過濾邏輯重複且難以維護的問題，建立了三種標準化工具，實作了「全域 (Global) > 牧區 (Zone) > 小組 (Group) > 本人 (Self)」的等級控制：

- **`applyScopeConstraints()`**: 用於 Repository 查詢前。自動根據 `userContext` 注入 `zoneId` 或 `groupId` 等查詢條件，確保資料在資料庫層級就被鎖定在合法範圍。
- **`assertScopeAccess()`**: 用於單筆資料存取。在 API 返回或更新特定 ID 前進行範圍校驗，若層級逾越則直接拋出 `403 Forbidden`。
- **`filterByScope()`**: 用於記憶體資料過濾。針對無法直接在 Query 層處理的跨表 Join 結果進行二次過濾。

### 2. Service 層級重構

- **`MemberService` 減重**
  - 移除了內部私有的 `applyScopeFilter` 與 `checkScopeAccess`。
  - 全面改用新的統一口徑工具，讓會友管理系統的權限邏輯符合「一次定義，隨處可用」原則。
- **`OrganizationService` 權限補強**
  - **結構查詢 (`getStructure`)**：現在會根據使用者的 Scope 自動剪枝。牧區長僅能看到所屬牧區，小組長僅能看到所屬小組，不再外洩全教會組織架構。
  - **成員查詢 (`getGroupMembers`, `getPendingMembers`)**：加入範圍檢查，防止透過直接呼叫 API 讀取非權限範圍內的人員名單。
- **`CourseClassService` 安全性提升**
  - **學員名單保護**：雖然班級是全域資源，但讀取班級學生詳情時，現在會與 `MemberService` 的 Scope 邏輯連動，確保名單顯示符合個資安全性。

### 3. API 與架構優化 (Service Layer Pattern)

- **領袖候選人 API 重構 (`GET /api/organization/leader-candidates`)**：
  - 將原本寫在 Handler 裡的混亂業務邏輯搬移至 `OrganizationService.getLeaderCandidates()`。
  - 加入了對應的 Data Scope 過濾，解決了過去可以跨牧區檢索領袖候選人的潛在隱憂。
- **Context 流轉優化**：
  - 更新了多個 API Handler（如 `pending-members.get.ts`, `structure.get.ts` 等），確保由 Middleware 注入的 `userContext` 正確傳遞至 Service 層，消滅了「管理員權限漏洞」。

### 4. 錯誤修復

- **修復 `CourseClassService` 匯入問題**：
  - 修正了重構過程中漏掉的 `applyScopeConstraints` 匯入，確保班級學員查詢功能運作正常。
